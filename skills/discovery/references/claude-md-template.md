# CLAUDE.md template for data science projects

Discovery Stage 6 emits this at the project root. It sets behavioral guardrails
for any LLM working in the project — even without the EDS plugin loaded.

Merge with project-specific instructions. The EDS plugin adds its own skills
on top; this file is the safety net that works standalone.

```markdown
# CLAUDE.md

Project-level behavioral guidelines for LLM-assisted data science work.
Generated by EDS discovery — edit freely, this is YOUR project's file.

## 1. Think Before Coding

Don't assume. Don't hide confusion. Surface tradeoffs.

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- Before reaching for a model, ask: does a rule, a threshold, or a SQL query already answer this?

## 2. Simplicity First

Minimum code that solves the problem. Nothing speculative.

- No features beyond what was asked. No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- If you write 200 lines and it could be 50, rewrite it.
- A heuristic that captures 90% of a model's value at 5% of the cost wins.

## 3. Surgical Changes

Touch only what you must. Clean up only your own mess.

- Don't "improve" adjacent code, comments, or formatting.
- Match existing style, even if you'd do it differently.
- Remove imports/variables YOUR changes made unused. Don't remove pre-existing dead code.
- Every changed line should trace directly to the request.

## 4. Goal-Driven Execution

Define success criteria. Loop until verified.

- "Add validation" → write tests for invalid inputs, then make them pass.
- "Fix the bug" → write a test that reproduces it, then make it pass.
- For multi-step tasks, state a brief plan with verify steps.

## 5. Data Science Discipline

These are non-negotiable regardless of project size or urgency.

### Audit before analyze
- Check grain, nulls, duplicates, ranges, and join cardinality BEFORE any analysis.
- After every join, assert row count — a fan-out join silently doubles your results.
- Log what `dropna()` removes. Silent data loss is invisible bias.

### Time flows one way
- Never shuffle time-indexed data in train/test splits. Use temporal splits.
- Never fit a transformer (scaler, encoder) before splitting — fit on train only.
- Every feature must be available AT DECISION TIME, not just in the training table.
- If a feature correlates > 0.95 with the target, it's probably leaking the future.

### Baseline is the burden of proof
- Measure a simple baseline (majority class, mean predictor, logistic regression) BEFORE any complex model.
- State the bar explicitly: "must beat AUC 0.71 by a margin that survives bootstrap comparison."
- If the baseline meets the stated success bar, stop. Ship it. Don't model for sport.

### Reproduce or it didn't happen
- Every stochastic call gets `random_state=42` (or the project seed).
- Pin package versions. Record data snapshots or hashes.
- One command should rerun the entire analysis from raw data to final output.

### Models serve decisions, not metrics
- A recall improvement that triples the review queue past capacity is a regression, not a win.
- Calibrate probabilities BEFORE setting any threshold.
- Thresholds are set by expected value and operational capacity, never by eyeballing an ROC curve.
- Report the metric AND the resulting operational load in the same breath.

### Honesty about uncertainty
- State sample sizes. Provide intervals where they matter.
- "Better" needs a confidence statement, not a point estimate.
- Multiple comparisons: if you tested 20 features, some will look significant by chance.

## 6. Project State

This project uses EDS. Key artifacts:

- `.eds/BRIEF.md` — the problem contract (decision, cost of being wrong, success metric, Plan)
- `.eds/PROJECT.md` — how to work in THIS project (data access, environment, conventions)
- `.eds/data-manifest.json` — registered data sources with freshness tracking
- `.eds/verification/` — gate pass/fail records (no green gate, no done)

Read `.eds/BRIEF.md` before starting any analysis. It is the ground truth for what problem we're solving and where we are in the lifecycle.

## 7. Anti-Patterns to Catch

Flag these immediately — don't wait for review:

- `accuracy_score` as the primary metric (almost never appropriate)
- `train_test_split` without `random_state` or with `shuffle=True` on time-indexed data
- Evaluating on the training set ("model.score(X_train, y_train)")
- `SELECT *` in SQL (no grain contract, schema changes break silently)
- Adding features without a hypothesis for WHY they'd help
- Setting a threshold by looking at test-set performance
- Reporting a metric improvement without a confidence interval
```
